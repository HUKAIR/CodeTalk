# Issue Tracker: GitHub

Issues and product specifications for this repository live in GitHub Issues. Use
the `gh` CLI from this checkout so the repository is inferred from `origin`.

## Public-data gate

This is a public repository. Before creating or updating an issue, comment, or
attachment:

- remove credentials, tokens, personal data, private repository content, and
  private filesystem paths;
- omit confidential product strategy, customer details, and unpublished data;
- use synthetic or already-public examples when evidence is needed;
- stop before publishing when the material cannot be confidently made public.

The same gate applies to command output pasted into an issue. Redact locally
before calling `gh`; do not publish first and edit afterward.

## Conventions

- Create an issue with `gh issue create`.
- Read an issue and its comments with `gh issue view <number> --comments`.
- List work with `gh issue list`, scoped by state and label.
- Update labels or assignees with `gh issue edit`.
- Close completed work with `gh issue close` and a concise verification note.
- Use native GitHub issue dependencies where available. Otherwise, put a
  `Blocked by: #<number>` line at the top of the issue body.

## Pull requests as a triage surface

**No.** External pull requests remain subject to manual review and are not fed
through the issue-triage state machine.

## Skill mappings

- "Publish to the issue tracker" means create a GitHub issue.
- "Fetch the relevant ticket" means read the GitHub issue and its comments.
- `/to-spec` and `/to-tickets` may create issues only after the public-data gate
  has passed.
- `/triage` processes incoming issues, not tickets generated by `/to-tickets`.
