# 产品技术问答(2026-06-23)

四个问题:多 agent 并行分析 + 对抗式核查后的结论(核查订正已并入)。
方法:4 路独立 subagent 各答一题(接地源码/文档/官方 spec)→ 对 MCP 事实、
「完全解决」声称、设计合红线 三项对抗核查。

---

## ① vibetrace 的「网站」是什么

**没有对外营销官网。** vibetrace 是本地优先 CLI(`pip install -e .` 得到 `vibetrace` 命令),
红线是「数据不出本机、禁 Web 框架」。所谓「网页」其实是它**在本机起的本地视图**(全绑
`127.0.0.1`、单文件离线 HTML、`--serve` 才开浏览器、落盘前脱敏):

| 视图 | 命令 | LLM |
|---|---|---|
| console 控制台(四视图单页,点击钻取) | `vibetrace console [--serve]` | 零 LLM |
| tunnel 时光轴(提交线性时间线) | `vibetrace tunnel [--serve]` | 零 LLM |
| report 汇报(当前仓快照) | `vibetrace report [--serve]` | 零 LLM |
| course 演进课程(分章+测验) | `vibetrace course` | 要 LLM(无 key 降级) |

4 个里只有 course 调 LLM;其余 3 个关掉大模型照样跑。本地服务每次随机选 127.0.0.1 端口
并打印,**只在命令运行期间存活**(没有固定网址)。

**官网首屏定位(N=3 校正后)**:把首屏从「记决策/认知层」翻成用户真有 pull 的一句——
> 问任何一行代码「当初为什么这么写」,用真实的 commit / 测试 / PR / AI 会话原话作答——
> 而不是让大模型事后编一个理由。关掉大模型也能用。

三支柱:入口=「这段代码当初为什么」· 护城河=零-LLM 确定性接地对抗 AI 反推式编造 · 本地优先零摩擦。

## ② MCP 是否支持不同 / 本地大模型接入

**核心:MCP 模型无关 —— 模型住在客户端(host),不住在 server。**(经官方 spec 逐句核实)

- **不同大模型?支持。** server 只通过 JSON-RPC 暴露 tools/resources/prompts,不感知调用方模型。
  任何实现 MCP 的客户端(Claude Desktop / Cursor / Windsurf / Cline / Continue…),背后接
  Claude/GPT/Gemini 都能连同一个 server。(spec 的 Sampling 机制专门让 server「保持 model-independent、
  不内置模型 SDK」。)
- **本地模型?支持,但看客户端。** 只要本地模型前端实现了 MCP client(Cline/Continue/LM Studio/
  带 MCP 的 Ollama 前端),就能一边跑本地模型一边调 MCP server(stdio 同机直连、不出网)。
  **瓶颈在「那个客户端有没有实现 MCP」,不在模型也不在 server**;本地工具对 MCP 支持参差。
- **vibetrace 自己的 MCP server**:纯 stdlib stdio、模型无关,暴露 ask/blame/graph/search。
  **区分两类模型**:`ask` 会调 vibetrace **内部配置的** LLM(`~/.vibetrace/config.json`,支持
  deepseek/openai/qwen/anthropic **四家全云端**)做综合;blame/graph/search 零 LLM。这跟「连它的
  客户端用哪个模型」是两码事。vibetrace 走自带内部 LLM,而非 MCP sampling 向 host 借模型。

来源:modelcontextprotocol.io 的 architecture / specification。

## ③ 不接大模型:安全性 + 能否完全解决三问卷

**安全性:零-LLM 是最硬一档,但 ≠ 零风险。**
它把现状最大的隐私采纳阻力——会话+diff 出网(默认 deepseek 云端)——整条砍掉,正中问卷
「客户代码不出本机」(2/3)。但消除的是「出网面」,不是全部。仍有本机面:① cache.db/vault
把分散的「为什么」聚合成明文集中文件(失窃后果更大)② cache.db 无加密、会被全盘备份带走
③ 误配:配过一次 key 跑 enrich,数据已出网且落进不可变缓存——而**当前无 `--no-llm`/本地
provider 硬开关(真实缺口)** ④ 脱敏只防 secret 模式,不防业务机密/PII/客户名 ⑤ usage.log
记了项目绝对路径(可能含客户名)。→ 对外只能说「数据不离开你电脑」,**不能升格「零风险」**。

**能完全解决吗?诚实结论:不能,「完全解决」是假命题。**
- **零-LLM 命中(护城河靶心)**:对抗 AI 编造(机制上不重述/反推)、隐私出网阻力、grep 盲搜→FTS、
  决策影响图。
- **结构上需 LLM、零-LLM 只能降级**:`ask` 综合答(无 key 降级为「带 SHA 原始材料自己读」)、
  周报/digest 可读叙事、course。能用、接地、不编、不出网,但不够顺——而这群用户对糙体验零容忍
  (留存=N=0 待验证)。
- **隐藏天花板**:旗舰痛「找回 why」(3/3)的高质量接地隐含依赖 enrich(要 LLM)把会话原话写进
  cache;纯零-LLM 下退化为只读 git 面包屑,而面包屑覆盖率常为 0。
- 用户各自头号痛(文件编码/需求变更/硬件固件)本就超域,用不用 LLM 都不做。

一句话:零-LLM 是「护城河的纯净结晶 + 隐私的最硬形态」,是**地基/差异化首屏**,不是完整产品。

## ④ 「指令回看」功能设计(已实现)

**关键洞察:数据层已存在,这是「加一个视图」不是造数据。** sessions/cursor_sessions 早就在抓
用户原始 prompt(每会话 `prompts` 列表,采集即脱敏,带 session_id/start/end/files_written/title/source)。

**已实现 `vibetrace prompts [--since "1 day ago"] [--source claude|cursor|both]`**(零 LLM):
按天/会话时间线列出你发给 AI 的指令 + 这会话改过的文件。复用 `scan_sessions`/`digest._since_to_dt`/
`_sources`/`align.align`/`redact_secrets`,新增 `prompts_view.py`(<100 行)。

**对抗审查定的红线(已落实)**:commit 关联用 align 是**软对齐**(±30min 时间窗 + 文件交集),
回看「今天」时大量工作还没 commit、长会话会乱牵、多对多。所以:
- v1 核心 = **确定性指令时间线 + files_written**(零猜测,仓内显相对路径、仓外只留文件名防泄露)。
- commit 关联 = **软对齐弱提示**,显式标「按时间+文件推测,可能不准」,**绝不渲染「✓已提交」**——
  软对齐冒充因果会砸 vibetrace「对抗反推式幻觉」的招牌。
- 数据形状更正:Cursor bubble 自带 `createdAt`(被丢弃),Claude 侧无 per-prompt 时间戳;v1 都不依赖。
- v2:接 evidence 锚点做「指令→AI 当时怎么回应→落成哪个 commit」三段接地;console 第五视图。
