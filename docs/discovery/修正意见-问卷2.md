# vibetrace 产品方向修正意见(基于已复审的问卷2 · N=2 暂定)

> **依据**:`docs/discovery/gap-analysis-问卷2.md`(已过 4 视角对抗复审:证据纪律 / scope 纪律 / 护城河-交叉引用 / per-tool,判 minor-revision,high/medium 问题已全部改进)。
> **效力**:全部为 **N=2 暂定**。N=2 **不是人群**;凡「2/2 复现」读作「双样本复现、待 ≥3 升级」。升为产品定论前须等问卷3-N + 过五道门。
> **与问卷1 关系**:多数痛点 2/2 复现(证据等级由 N=1 升至 N=2,但仍 <3 不升人群);一处关键 **contradict(主力工具)**;两条新机制(AI 搬运丢注释、靠近代码留痕认知)。

## 一、该加强的(双样本复现 → 信心提升,但仍守 ≥3 门槛)

1. **核心叙事:零-LLM 接地对抗反推编造**。问卷2 贡献护城河的**实测被坑事件**(贴回 Claude 编「内存优化」、真因「流式响应不断连」、损失 2h、方向走偏)。
   - **加强**:把「blame/ask 引真实会话原话/SHA vs LLM 重解释」做成**对位演示**作差异化首屏。
   - **证据纪律**:这是 **N=1 行为级实证**;与问卷1「只信 6 分」(态度评分)合起来是「2/2 复现态度警惕 + 1 个实测事件」,**不是两个等强被坑事件**。对外文案不可暗示「两人都被坑过」。
   - **诚实边界(必带)**:演示须标明护城河**有条件失效**——会话被清理时(问卷四 Q3/Q7 已发生)零-LLM 接地无源可接,与 LLM 重解释拉不开差距。

2. **以 ask/graph 为 wedge 服务「找回 AI why」**(O1)。问卷2 主力 = Claude Code = 主线源,**主力源对位是必要条件**。
   - **证据纪律(纠过满)**:「核心价值不落空/零迁移/首批早期采纳画像」是 **N=0 假设**(他从未试用 vibetrace),不可写进定论。dogfood 胶囊直接锚定他的 5 个 why 案例,**预期仅 2 例会话可命中**(loader.go/IMRead)、1 例真因在原型注释出域(2048)、2 例无源(CLAHE 无处可考、Flask 会话已删)→ 实测命中率再定 P0。

3. **会话定向接地替代 grep 盲搜**(O3,与 O1 同 wedge,去重看待)。
   - **口径修正**:省时目标锚 **grep 子步 12min**,不是端到端 20min(后者含问硬件负责人 ~5min + 翻原型注释,非全可省);且 2048 真因部分在原型注释,会话接地不一定命中。

4. **手工留痕必崩 = 痛点 2/2 复现**(决策日志 3 周即弃 / 博客14月+commit规范3月全断)。
   - **拆档(纠偏换)**:**痛点 有证据 N=2**;但「agent-seed 写时捕获被接受」是 **解法 N=0 假设**,且他对「需额外动作的留痕/工具」**弃用率 100%**(ADR 3 周、Notion 2 周、claude-history-export 周末即弃)。**不得标「强项」**,沿用问卷1 已确立「解法是推断、非明示需求」。

5. **胶囊回面绑事故证据 + 指标改回面后处理率**(O6)。N=2 同根因确认:TODO 失效是**优先级博弈非可发现性**。问卷2 真案例(QA 报 bug、OOM 报警才回来)证明只有附「已/将成事故」证据才撬得动。

6. **持久化载体方向(commit/注释靠近代码才活)弱支持 commit-trailer 落点**。
   - **降级(纠确认偏误)**:用户认可的是「**人写的注释 + commit 正文**」持久;**未直接背书产品「自动写入 trailer / SHA 为键缓存」机制**(他认可的是自己手写的正文,与其对额外动作零容忍隐含张力)→ 仅作方向弱支持,机制接受度待验证,不称「教科书案例/强力背书」。

## 二、该收敛的(N=2 下不要过度声称)

- **「证伪单一主力假设」→ 降为「1 个反例」**:N=2 各 1 只能说「出现 1 个反例、提示不能默认单一主力」,**不足以证伪任何人群级分布**(完全可能某主力占多数)。这与本目录「同痛 ≥3 才升人群」硬规则一致。
- **付费意向 → 条件式弱信号 N=1**:问卷六 Q6「会认真考虑」非实际付费(他实付只有 Claude Pro 且明说不为存档决策),且挂在尚未对位的周报形态上 → 记「待验证变现假设」,不称「付费意向锚点/最接近付费的信号」。
- **隐私 N=2 未复现 ≠ 人群级中性**:问卷2 无顾虑可能源于主力本地 Claude Code **未触发出网场景**,不能与问卷1 强顾虑**对消**;列开放项,待问卷3 验证(尤其问中云端主力受访者)。
- **single-event 不重复计权**:2048/CLAHE 是同一组 N=1 事件,别在 gap 矩阵里支撑两条并把一条抬为「最贵痛点·高」;「最贵」改「他主观最懊恼的一次(N=1)」。
- **severity/priority 标口径**:N=2 下 severity 仅反映痛感强度;P0/P1 = **验证优先级**非建设优先级;建设类升 Now/Next 须 ≥3 复现 + 可复算成本 + 过五道门。

## 四、capture-source 分裂对「不分版」结论的影响

> **措辞校正**:仓内**无成文「不分版」doctrine**(README per-tool 行指向 SKILL/PM brief;SKILL 门3 把 Cursor vs Claude Code 列为**可证伪假设**、门4 列竞品;无 PM brief 文件)。故「不分版/双源并行」是**分析师采用的工作假设**,不以「强化现行结论」把假设升格为定论。

1. **N=2 实测主力工具相反(Claude Code 1 / Cursor 1)= 1 个反例**:提示不能默认单一主力,但**不构成分版触发条件**(触发仍需 N≥3 的结构性需求差异**或**接地质量差异)。

2. **对工作假设的影响 = 不松动、但要把两点显式化**:
   - **(a) 双源必须都做实,但默认不对等**:Cursor 源已 merged 入 main,**但 opt-in 默认关闭**(`config.py:22` `sources=["claude"]`)。问卷2(主力 Claude Code)默认即读到主力源;**主力 Cursor 用户(问卷1)默认配置下读不到自己的会话,需手动 opt-in** → 「双源并行已做实」**不等于「开箱对等覆盖」**,文案/onboarding 须诚实。
   - **(b) 接地质量非对称,与样本量无关(纠「推给 N≥3 远期」)**:Cursor `files_written` 早期取自上下文文件(relevantFiles/attachedCodeChunks)→ 把错 why 以 high 置信「漂白」进不可变 SHA(#30 专门修);composer↔commit 对齐**比 Claude 松**;composerData schema **非官方、随版本变**(见 grounding design spec)。**这是客观事实,不依赖更多样本** → 结论改「双源并行但**接地质量非对称、Cursor 需更强置信度护栏**」,**不可读成「统一就行/双源无差别」**。

3. **会话价值链脆弱性双样本确认**:两源会话都「留不住」(Cursor 被覆盖/物理丢失;Claude Code 难查 + 偶发物理丢失且会过期)。两人都依赖「靠近代码的留痕」。→ 把会话原话蒸馏进 commit trailer / SHA 为键缓存的**落点方向**得到双样本方向支持,**但具体自动机制接受度仍待验证**(见 §一.6)。
