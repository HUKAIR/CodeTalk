# 护城河对照卡:真实记录 vs 纯 diff 反推

**一句话**:让只看 diff 的 AI 反推「当初为什么这么改」,它会**自信、流畅、却编造或漏掉真正的 why**——尤其「为什么**不**那么做 / 战略取舍 / 预算约束」这类**结构上不在 diff 里**的理由。CodeTalk 引逐字真实记录(commit 面包屑),不推断,所以不会编。

**这正是「6 分信任天花板」的成因**:用户只信 AI 重解释约 6/10,因为它从 diff 反推会编。对照卡把这件事**可复跑地**摆出来。

---

## 方法(公平起见)
- 取 CodeTalk 带真实 `Vibe-Decision`/`Vibe-Rejected` 面包屑的 commit,把**脱敏后、仅代码段**的 diff(无 message、无注释、无上下文、**已剔文档段**)喂 LLM 反推 why,跟 CodeTalk 零-LLM 引的真实面包屑并排。
- 自动挑「泄漏最低」(why 字面不在 diff)的样本,避免拿「diff 已夹带 why」的 commit 反向坑自己。
- 复跑:`python3 scripts/blind_test.py . 5`。判对错由人(语义需人,脚本不自动打分)。

## 结论
- 纯 diff 反推 = git 考古工具(Cursor / Claude Code / Codex 的 git skill)能给的极限——它只能从 diff 反推,正是用户只信 6 分处。
- CodeTalk 引**逐字真实 commit 记录**,不推断 → 不编;对照卡把这个差异摆成可核验、可复跑的证据。

## 反面钩子:「数据还在,why 却接不回」(接地的反面 · 可落地页/受访者破冰)
真实案例(`forum.cursor.com/t/cursor-lost-all-my-agent-chat-history/158837`,已核):Cursor 用户**一次异常关闭后**,~175 条 agent 对话在历史面板**全不显示**——但**数据其实还在 `state.vscdb`**(`composer.composerHeaders` 175 条名字俱全、本地 transcript 文件夹也都对得上),只是每条 header 的 `workspace` 被置成 `null`,历史面板**全部过滤掉**、接不回了(Cursor 团队成员已在帖中确认此诊断)。
→ **这正是「接地」的反面**:把 why 寄存在易损的会话 DB / UI 状态里,换个 workspace、重启、隔几周回来就接不回。**CodeTalk 反着来**——why 钉死在 **immutable commit / PR / 会话逐字真实记录**(以 SHA 为键、永不重算;SHA = git 给每次提交的唯一指纹码,`git show <SHA>` 就能原封不动调出当时那次改动的原话),不依赖任何 UI 状态;关掉工具、换机器、隔三周回来,记录还在原处、逐字可核验。一句话:**别人的「为什么」躺在会丢的聊天框里;CodeTalk 的「为什么」长在 git 里。**

## 诚实边界(别 overclaim)
- 这是 **dogfood 证据(N=1 仓 CodeTalk、作者自判)**,**不是「天花板已破」**。
- 「破 6 分天花板」的硬证据 = 把这张卡**给真实受访者看、测其信任分是否上移**——本卡是那一步的**素材**,不是结论。
- 泄漏标只测「why 字面是否落在 diff」,不测「能否推断」;对短/改写 why 偏向「未夹带」(偏利好护城河),读时打折。
- **信任位移仅 N=1、且受访者为作者本人(展示时由作者措辞引导)**:dogfood 中作者看卡后对「AI 解释 why」的信任 8→10 成,明说被「逐字原话 + 能自己核对」打动 → 仅**弱确认卡的两个杠杆方向选对**;但作者**起点已 8 成(非 6 分基线)**、属内部视角且有引导,**这不证「破 6 分」**。破 6 分的硬证据仍需对 **≥3 名外部受访者**跑「LLM 凭空重述 vs 引真实 SHA/原话」对比 demo、先测基线再看卡。
