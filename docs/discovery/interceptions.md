# 拦截记录 · interceptions

**这里记录 CodeTalk 是否真的阻止了一次理由丢失型踩坑。** 在出现可核验记录前,
产品只描述其证据呈现能力,不宣称已经阻止过真实事故。

## 什么算一次「拦截」(可证伪判据,人判)
全部满足才算,缺一不算:
1. **接地先发**:`codetalk review` / `blame` / `ask` 在改动落地前打出了某条逐字
   `Vibe-Rejected`(或 `Vibe-Decision`)的真实 why + SHA;
2. **命中重引入**:当前改动确实在把那条「曾放弃的方案」重新引入(同文件/同概念);
3. **改向**:你因此撤回 / 改方案 / 标记已知——即真避免了一次踩坑。

> 诚实边界:第 2 步「同概念重引入」是**人判**,不是工具自动判定——散文 `Vibe-Rejected`
> 对代码 diff 的语义匹配需模型(R6 钉死不做)。`review` 只确定性地把命中的否决项提到眼前
> (顶部「⚠ 拦截检查」),由你确认。

## 记一条(每次满足上述三条就追加)
```
### YYYY-MM-DD
- 丢失的 why:<曾否决/曾决定的原话> (`<sha7>`)
- 哪条命令 surface:codetalk review / blame / ask
- 本要做的错事:<没有 codetalk 的话你正要重引入什么>
- 实际避免的后果:<省下的返工 / 拦下的事故>
```
