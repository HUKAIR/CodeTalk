# vibetrace 外部对标 · 第七轮(AI 编码 agent 的「计划-执行一致性 / 意图-结果差距」评估)

日期:2026-06-27 · 方法:`/deep-research`(103 agents,5 角度;部分 verify 因 claude.ai 月度上限掐断,综合与强证据 finding 已成形)· 三视角 · 纯外部对标。
**裁决:有清晰可建功能(比 R6 强)。** 红线内、零-LLM、无竞品占据的可守缝 = **「AI 声称 vs git 实际」确定性偏差报告**(复用现成 `sessions.files_written`+git+align,报**字面可数缺口**,非语义「完成度 70%」)。

> 不重复前几轮(多模型对抗辩论无增益、完备幻觉检测需模型、MCP=集成、local-first 非护城河、本地模型便利非对等、营收基准不可得)。对位用户原始痛点:**AI 说要做却没做/没做全**。

## 一、技术现状(强证据)
- **部分完成度评估是成熟、peer-reviewed 概念,别宣称「没人测部分完成」** `[强]`:AgentBoard「Progress Rate」(完成子目标占比)、OctoBench「Check-item Success Rate」(满足检查项比例给部分分)、SWE-EVO「Fix Rate」(修复 FAIL_TO_PASS 测试占比)——均针对「binary pass/fail(SWE-bench Resolved Rate 全或无)藏住部分进度」而设。(arXiv 2401.13178 / 2601.10343 / 2512.18470)
- **「意图-结果差距」框架被外部验证** `[强]`:agent 会「看着对、却悄悄违反约束」;过程/步骤级 trajectory 评估是揭示它的成熟手段。OctoBench 专为「解决任务」vs「遵守规则」解耦而建。

## 二、护城河核心:确定性 proxy 可行,但都靠预置 oracle(vibetrace 没有)
- **确定性零-LLM 部分完成度 proxy 确有先例,但只经 test-as-spec / state-as-spec 对预置 oracle 执行,从不靠语义判断** `[强]`:SWE-EVO Fix Rate + 回归门、Agent-Diff before/after 结构态差 + 声明式断言、SWT-Bench Fail-to-Pass 合取。(2512.18470 / agentdiff.dev / swtbench.com)
- **⚠ ORACLE GAP(对 vibetrace 的关键边界)** `[强]`:上述每个确定性先例都**因预置 ground-truth oracle**(golden patch / 已知 FAIL_TO_PASS·PASS_TO_PASS 集 / 人写断言)才确定。**vibetrace 没有 oracle——只有 AI 自报意图 vs git diff**。故这些先例**证明「确定性部分打分」这个一般模式可行,但不能直接移植**到 vibetrace「无 oracle、自报 vs 实际」的集合比对。vibetrace 可守的确定性动作是更窄的「自报范围 vs 实际 diff」集合差(声称文件 vs 触碰文件;声称加测试 vs 有无 test 文件;声称改 vs 空 diff)——高精度低召回、**不能测语义「完成意图的 X%」**。

## 三、诚实张力(R6 回声,确认)
- **语义 adherence/意图-结果判断被业界视为需 LLM,确定性零-LLM 打分是公认开放空白** `[强]`:连 OctoBench——刻意编了 7098 条「客观可判」检查项——实际打分仍回退 3 模型 LLM-judge 面板,并把「更强的确定性检查」列为未解决 future work。→ **任何措辞成语义完成度(「AI 完成了计划的 70%」)的 vibetrace claim 都是确定性无法诚实支撑的**;诚实形式是字面可数缺口(「AI 说要动 N 文件、实际动 M、P 个没碰;说加测试、未见 test;说改、diff 空」)。

## 四、竞品门(空白缝)
- **最近竞品 Agentplane(local-first / MIT / CLI / 活跃)在 repo 内存 task intent + approved plan + checks + traces + Agent Change Record,但不提供任何「量化 plan adherence / 完成度 / 意图-结果偏差」机制** `[强]`。→ **确定性偏差度量(尤其无 oracle、复用既有「会话声称 ↔ git 实际」对齐)这角度基本无人占据。**(agentplane.org / github basilisk-labs/agentplane)

## 五、pm 五门裁决 + 落点
- **护城河门**:零-LLM 确定性「声称 vs 实际」偏差,复用既有会话↔git 对齐,撞不到 LLM-judge agent-eval 满地腹地——可守空白缝。
- **M0 红线门**:仅 stdlib + 现成 `sessions`/git/`align`;零 LLM;数据不出本机。红线内。
- **证据门(N=1)**:外部强证据支撑「部分完成度是真概念」「确定性需 oracle」「语义需模型」「竞品未占偏差度量」;但「vibetrace 偏差报告有用户价值」无外部直证(对位 N=1 痛点强、人群结论需 ≥3)。
- **竞品门**:Agentplane 占 audit-trail 不占偏差度量 → 空白。
- **北极星门**:服务「防 AI 说了没做型踩坑」;但「偏差数」是度量产出,北极星仍是回面后处理率,偏差报告作护栏/防踩坑信号。

**裁决:可建。** 红线内最小做法 = **`sessions.files_written`(AI 声称)∩/∖ git diff 文件(实际)** 的确定性偏差报告:
- 「AI 声称动 N 文件,实际 M,P 个**声称却没碰**」(最硬信号:说了没做)。
- 「声称加/改测试,但 diff 无 `test_*`/`*_test` 文件」。
- 「声称改某文件,但该文件 diff 为空/未在提交」。
- **诚实边界(必守)**:只报**字面可数缺口**,**禁**「完成度 X%」「设计可行吗」「执行质量」这类语义断言(需模型,违零-LLM 护城河)。措辞钉「声称 N、实际 M、P 个没碰」。

## 六、诚实边界 / 被否决强 claim
- 被对抗**否决**:① 「现有框架只测 binary success」(1-2 否——部分完成度 metric 确存在,别说「没人测部分完成」);② 「Agentplane 的 ACR 已做文件级 adherence 联动」(1-2 否——它记 changed files + 跑 checks 但**不算**planned-vs-actual 偏差);③ 若干 git-ai / OctoBench「Stability Trap」claim 0-0 未验证,不依赖。
- 时间敏感:核心源 2026-01~05(OctoBench/SWE-EVO/Agent-Diff/Agentplane),指标定义数周内可变;部分 verify 因月度上限未跑完,综合基于存活强证据。
