**① 信任天花板 —— 二手最硬(人群级量化)**
- **SO 2025**(N=33,244,67.8% 回收率):highly-trust 仅 **3.1%**;distrust **45.7%** > trust **32.7%**;10+ 年经验者 highly-trust **2.5%**、highly-distrust **20.7%**(资深更不信)。`[survey 强]` survey.stackoverflow.co/2025/ai
- **SO 信任趋势(2023→2025)** :正面态度 70%+(2023/2024)→**60%**(2025);信任 ~43%(2024)→**29%**(2025,跌 11pp);采用率反升到 **84%**——**信任↓采用↑的悖论**。`[survey 强]` stackoverflow.blog/2026/02/18/closing-the-developer-ai-trust-gap/
- **SO 2025 最大痛点**:66% 头号挫败=「AI solutions **almost right but not quite**」;**75.3%** 不信 AI 答案时**转去问真人**。`[survey 强]` leaddev.com/technical-direction/trust-in-ai-coding-tools-is-plummeting
- **SO 2024**:遇困难头号问题逐字「Don't trust the output」66.2%。`[survey 强]`
- **DORA 2024**:39% 对 GenAI 输出质量只「a little / not at all」信任(页未披露 N)。`[survey 中强]` dora.dev/research/ai/trust-in-ai
- **Sonar State of Code 2025**(N>1,100):96%「不完全信任」AI 代码、53%「appears correct but unreliable」(媒体二手转述)。`[survey 强/转述中]`
- **METR 2025 悖论**(受控实验):有经验开发者用 AI 实际完成任务 **慢 19%**,但自我评估以为**快 20%**——体感与实测反向。`[实验 强]` uvik.net/blog/ai-coding-assistant-statistics/
- **SO 2025 调试成本**:**45%** 开发者说调试 AI 代码**比人写的更耗时**;AI 代码 suggestion 接受率仅 **27–30%**、即 70% 被拒。`[survey 强]`
- **JetBrains 2025**(N=24,534):85% 常用 AI,但 SO 2025 仅 **16.3%** 认为 AI 让自己「大幅提效」、**41.4%** 认为「几乎没影响」。`[survey 强]`
→ 据此把 N=1 的「6 分」从主证据降为「我的深访也复现了人群结论」的佐证。人群信任只有 ~30%、且还在逐年下滑——**「6 分天花板」甚至偏乐观**。METR 悖论还揭示一个 ⑤ 旁证:开发者**高估了 AI 的帮助**(以为快 20% 实际慢 19%),在 why-解释上类推 = **可能高估了 AI 反推 why 的准确度**(我们 N=1 作者自测的 8 成信任或许正是这种过信)。

**② why-loss / 决策上下文痛(AI 时代加剧)**
- **Atlassian/Wakefield**(N=3,500、6 国):开发者头号时间杀手「finding information」(口径偏宽=泛化找信息,非专指 why-loss,外推谨慎)。`[survey 强,口径宽]`
- 学术行为观测:开发者 58–70% 时间在「理解现有代码」(Feitelson 2023 引 Minelli/Xia)。`[survey 强,上位证据]` arxiv 2410.21990
- 一手轶事(精准命中):Claude Code compaction 丢上下文,原话「the 'why' behind choices — gone」。`[blog 中]` dev.to/gonewx
- **Ghost Code(幽灵代码)**(概念提出 · 精准命中 ②+⑤):AI 生成的代码「was generated by AI, passed tests and reviews, entered production, but **cannot be traced back to an explicit human decision**」;通过 Inference Creep(AI 推断关联变更、diff 膨胀但连贯、reviewer 验结果不验决策)不知不觉进主线;「Git history gradually loses its explanatory power. What remains is **a timeline of execution — not a record of decisions**」;调试时「**debugging becomes a conversation with a ghost — a reasoning process without an author**」。`[blog 一手 中,概念清晰但无量化]` medium.com/@spark.tsai/ghost-code
- **认知债(Comprehension Debt)**(AWS dev.to):「Code that works. Code that **nobody understands**.」;「Debugging code you didn't write takes **3–5× longer**」;「there **was no reasoning**. There was **pattern matching** on training data」——AI 代码的 why 结构上不存在,不是「忘了」而是「压根没有过」。`[blog 一手 中]` dev.to/aws/the-hidden-cost-of-ai-coding
- **代码流失率翻倍**(GitClear,211M 行):code churn rate 从 **3.1%(2020)→5.7%(2024)**;复制粘贴率 8.3%→12.3%;5+ 行重复块**翻 8 倍**——AI 大量产出低归属代码、加速 why-loss。`[数据分析 中强]`
- **「The Code Nobody Understands」**(Monnerat 2026):「The code may be correct. It may even be elegant. But **the reasoning that produced it is not available to you**.」;「We are **generating code faster than we are generating that knowledge**. The gap is not visible in day-to-day metrics. It **will become visible under pressure**.」`[blog 一手 中]` davidmonnerat.com
- **「AI Coding Hangover」**(InfoWorld 2026):「They just **moved it, multiplied it**, and **buried it under layers of unmaintainable generated code**」;「Code that works until it doesn't」。`[InfoWorld 中强]`
- **arxiv 2603.28592《Debt Behind the AI Boom》**(大规模实证,**304,362 commits / 6,275 repos / 5 大 AI 工具**):484,606 个 issue 被引入;**15–29% 的 AI commit 至少引入一个 issue**;**24.2% 至今仍存活**;安全 issue 引入量**近 2× 修复量**;代码气味占 89.1%。`[学术 强,大样本]`
→ **AI 时代的 why-loss 不只是「忘了记」(问卷1/2/3 的痛),还有一层新痛:「AI 代码压根没有 why、从出生就是幽灵」**——vibetrace 的写时面包屑(Vibe-Decision)正打这层。arxiv 30 万 commit 实证已经量化了后果(24.2% issue 存活),下一步只差把 why-loss 跟 issue 存活率因果绑定。

**上下文退化研究(④ 的机制层 · 不是 bug 而是结构缺陷):**
- **Factory AI 研究**:agents 在长 session 中系统性遗忘——「Agents **forget which files they have modified**. They **lose track of what approaches they have already tried**. They **cannot recall the reasoning behind past decisions**. They forget the original **error messages** or technical details that motivated particular solutions.」`[研究 中强]` smarterarticles.co.uk
- **Sourcegraph Amp**:退役 compaction 功能,因「recursive summaries began to **distort earlier reasoning**」——多次压缩后前期决策被扭曲。`[工程博客 中]`
- **Chroma Labs**:模型在短 prompt 上 90%+ 准确,32k token 时**准确率骤降**;且**有逻辑组织的上下文反而比乱序文本表现更差**(反直觉)。`[研究 中]`
- **安全维度**:AI 代码 **45% 引入安全缺陷**(Java 最差 70%);仅 **3.8%** 开发者同时报告低幻觉率且有信心不审就发。`[研究 中强]`
- **O'Reilly Radar**(Mike Loukides 2026):亲历 Gemini compaction——「notes from the first half of the session were just… **gone**」;建议用 handoff document 而非 continuation prompt,因「important information discovered during discussion… if not written down, the team will have to have the discussion **all over again**」。`[O'Reilly 中强]`
- **开发者反馈**(CleanAim / dev.to 汇编):「After 2–5 prompts it **starts ignoring instructions**」;agent 像「a **junior developer with short-term memory loss**」;「AI agents have **no mechanism to carry knowledge between conversations**」。`[多源 blog 中]`
- **ADR 演进信号**:AGENTS.md(Codex/Claude Code 项目规则文件)开始补 ADR(Architecture Decision Records)的位——但面向 agent 不面向人;真正的 why(为什么这么决定)仍不在任何文件里。`[趋势观察]`
→ **④ 不只是 bug(issue 层)——agent 的上下文退化是结构性的**:session 越长、压缩越多、reasoning 越失真。O'Reilly 级别的作者也亲历「half gone」。这意味着即使 bug 全修了,**AI 自己也不记得自己当初为什么那么写**——零-LLM 确定性接地(不靠 AI 回忆、靠 immutable SHA)的价值正在于此。

- X 一手观点(通俗背书,KOL 热度≠发生率):「长对话越聊越笨,老消息被挤出 context window 就忘了」(NetworkChuck 类)。`[X 一手观点 弱-中,URL/原话 web 初采待核]`
- **公开真空不变**:无任何带 N 的调查专测「AI 编码对话历史丢失」→ 能说「跨 5 产品被 8+ issue 反复报告 + 结构性退化有研究支撑」,**不能说「X% 的人丢过」**。但密度 + 机制研究 = **从「碰巧被报告的 bug」升为「整个 LLM agent 架构的固有缺陷」**。

**⑤ 接地 vs 反推「差值」(vibetrace 命门)—— 公开几乎为零(社交检索后仍只补到「半边」)**
- 间接旁证:SO 2025 75.3%「不信 AI 答案时会转去问真人」(意向、且是「转向人」非「转向 git 原话」,降档);arxiv 2603.17150《Intent Formalization: A Grand Challenge》把「intent 无法可靠从代码反推」立为公开难题(仅 abstract)。`[中]`
- X 一手观点(只证半边):Simon Willison 转引并赞同——「不懂代码只能反复求 AI 修 = 用另一张信用卡还债」。`[X 一手观点 中,URL web 初采待核]` ——**只证「AI 解释/修复不可信」这半边,没说「所以我去翻 git 原话」(差值的另一半仍缺)**。
- **独立工具造者(行为级 · 有人真动手造了)**:
  - **Entire CLI**:把 AI session 完整归档进 git 隐藏分支(prompts / responses / tool calls / files touched),因「standard commit 之后 reasoning、decisions、tool calls — **all gone**」,6 个月后调试无法理解 intent。`[blog 一手 中]` dev.to/stevengonsalvez
  - **git bla(i)me**:把 agent conversation traces 存为 git notes 链到行级代码,因「When you look at a line in Git blame, you usually want the **reason** behind it, not a person」;traces 成为 **stable, auditable evidence**「linked to commits with stable identifiers」。`[blog 一手 中]` ronaldtebrake.nl/blog/git-blame-intent-aware-agents/
  - → 两个工具**独立把同一痛定义为问题并自建解决方案** = 行为级验证(比抱怨强);但**都没做到「零-LLM 确定性接地」**:Entire 存 session 文本但不按 SHA 索引不去 LLM 编、git bla(i)me 存 traces 但仍需 agent 后处理。**vibetrace 的缝仍在**。
- 其它 framing 旁证:第三方工具(`git-why`)独立把问题定义为「`git blame` 给你 who/when、给不了 why」——与 ⑤ 命题逐字同构,但是工具作者营销页、非开发者发言。
- **Ghost Code 佐证(② 条目,同时命中 ⑤)**:「there **was no reasoning**. There was **pattern matching** on training data」——AI 代码的 why **结构上不存在**;「debugging becomes a conversation with a **ghost** — a reasoning process without an author」。这不是「AI 忘了 why」,是「**AI 从未有过 why、它只有 pattern**」——对照卡的差值命题在这个角度有了机制解释:**不是 AI 反推不准,是 AI 根本没有东西可以反推,它只是在用 pattern 编一个看起来合理的故事**。`[blog 中,机制旁证]`
- **METR 过信推论(① 条目,间接命中 ⑤)**:开发者用 AI 实际**慢 19%** 但自以为**快 20%**——体感与实测反向 40pp。类推到 why-解释:开发者可能**系统性高估** AI 反推 why 的准确度(我们 N=1 作者的 8 成信任或许正是这种过信)。**如果过信是系统性的,那对照卡的价值不只是「让不信的人更信」,还有「让过信的人意识到自己在过信」**。`[实验→推论 弱,方向有力]`
- **真相更新**:仍无公开数据**直接量化**「翻 git/PR 原话 vs LLM 重述 why」的可信度差值。但现在有三层旁证支撑:(a) 2 个独立工具造者验证痛点存在;(b) Ghost Code 机制解释「AI 从没有过 why」;(c) METR 悖论提示开发者系统性过信 AI。**量化差值仍只能自己造数据**(对照卡 + C 段自有-commit + ≥3 真人验),但**叙事素材比第一轮厚了一个量级**。

## Vibe coding 反噬(② why-loss 在 AI 时代的放大器)

> 📌 下面数字多为聚合/二手转述,**「8000 家创业公司重建」未查到原始研究命名/方法论 = 不可外引的裸数字**,仅作背景感知。

- **arxiv 2512.11922**《Vibe Coding in Practice》:核心发现之一是「**lack of explicit design rationale**」= 生成 AI 代码技术债的关键上游。`[学术 中强]`
- **Rework/质量衰退**:重做率 +30–60% within 6 months;代码重复 up to 4×;DORA 指标 7.2% 交付稳定性下降(与 AI 采用相关)。`[Kyros blog,厂商二手 中]`

- **「月 3 断崖」**:多个来源描述 vibe-coded 项目在 ~3 个月后进入「加功能就坏已有功能」的崩塌循环。`[多源 blog 中,无严格量化]`
→ **Vibe coding 把 ② why-loss 从「个人偶发」推成「行业流行病」**:当 42% 代码是 AI 生成(SO 2025)、且绝大多数跳过「记 why」→ 3 个月后回头看的人面对的是**大片 Ghost Code + 零 design rationale**。这是 vibetrace 写时面包屑(Vibe-Decision)的**结构性机会窗口**——不是未来可能的痛,是**现在正在爆发的痛**。
