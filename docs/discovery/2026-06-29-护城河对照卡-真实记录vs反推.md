# 护城河对照卡:真实记录 vs 纯 diff 反推

**一句话**:让只看 diff 的 AI 反推「当初为什么这么改」,它会**自信、流畅、却编造或漏掉真正的 why**——尤其「为什么**不**那么做 / 战略取舍 / 预算约束」这类**结构上不在 diff 里**的理由。vibetrace 引逐字真实记录(commit 面包屑),不推断,所以不会编。

**这正是「6 分信任天花板」的成因**:用户只信 AI 重解释约 6/10,因为它从 diff 反推会编。对照卡把这件事**可复跑地**摆出来。

---

## 方法(公平起见)
- 取 CodeTalk 带真实 `Vibe-Decision`/`Vibe-Rejected` 面包屑的 commit,把**脱敏后、仅代码段**的 diff(无 message、无注释、无上下文、**已剔文档段**)喂 LLM 反推 why,跟 vibetrace 零-LLM 引的真实面包屑并排。
- 自动挑「泄漏最低」(why 字面不在 diff)的样本,避免拿「diff 已夹带 why」的 commit 反向坑自己。
- 复跑:`python3 scripts/blind_test.py . 5`。判对错由人(语义需人,脚本不自动打分)。

## 结论
- 纯 diff 反推 = git 考古工具(Cursor / Claude Code / Codex 的 git skill)能给的极限——它只能从 diff 反推,正是用户只信 6 分处。
- vibetrace 引**逐字真实 commit 记录**,不推断 → 不编;对照卡把这个差异摆成可核验、可复跑的证据。

## 诚实边界(别 overclaim)
- 这是 **dogfood 证据(N=1 仓 CodeTalk、作者自判)**,**不是「天花板已破」**。
- 「破 6 分天花板」的硬证据 = 把这张卡**给真实受访者看、测其信任分是否上移**(pm-brief 的 G2 原义)——本卡是那一步的**素材**,不是结论。
- 泄漏标只测「why 字面是否落在 diff」,不测「能否推断」;对短/改写 why 偏向「未夹带」(偏利好护城河),读时打折。
